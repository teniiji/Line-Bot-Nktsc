# Line-Bot-Nktsc
ไลน์ OA ตอบคำถามสมาชิกสหกรณ์ออมทรัพย์ครูหนองคาย จำกัด
(Next.js App Router + TypeScript + Prisma/PostgreSQL + Claude API)

## 🤖 ระบบการทำงานหลัก (System Architecture)
บอทรับ Webhook จาก LINE -> เรียกใช้งาน Claude (Tool-use Agent) ใน `lib/financeAgent.ts` (ตัว runner — ส่วนประกอบแยกอยู่ใน `lib/agent/`: types, state, tools, prompts, forwarding, handlers) เพื่อประมวลผล:
1. **บันทึกรายการ:** ตรวจสอบสลิป (ยอดตรง, บัญชีปลายทางต้องเป็นสหกรณ์ — มีชั้นตรวจซ้ำในโค้ด `lib/recipientCheck.ts` ปฏิเสธสลิปที่ผู้รับเป็นบุคคลธรรมดาเสมอไม่พึ่งดุลยพินิจโมเดล, ป้องกันสลิปซ้ำด้วย SHA-256) และแยกประเภทรายการ (ฝากเงิน, ซื้อหุ้น, ชำระหนี้) — ธุรกรรม "ชำระหนี้" ต้องระบุประเภทเงินกู้เพิ่ม และ "ฝากเงิน" ต้องระบุเลขที่บัญชีที่จะฝากเพิ่ม ก่อนจะบันทึกจริง — ถ้าชื่อในช่อง "จาก" ของสลิปไม่ตรงกับชื่อสมาชิกที่ลงทะเบียนไว้ (`lib/nameMatch.ts`, เทียบแบบยืดหยุ่น ไม่นับคำนำหน้า/เว้นวรรค) ต้องให้สมาชิกยืนยันก่อนด้วยว่าเป็นธุรกรรมของตัวเองจริง — รับสลิป/เอกสารประกอบทั้งแบบรูปภาพและไฟล์ PDF (ไฟล์ประเภทอื่นที่ไม่ใช่ PDF จะได้รับแจ้งว่าไม่รองรับ) เก็บ `slipIsPdf`/`imageIsPdf` ไว้เพื่อให้การส่งต่อรู้ว่าต้องส่งเป็นข้อความลิงก์แทนข้อความรูปภาพ เพราะ LINE Messaging API ส่งไฟล์แนบให้เจ้าหน้าที่ไม่ได้ ส่งได้แค่รูปภาพเท่านั้น
2. **ยืนยันตัวตน:** เทียบข้อมูลกับฐานข้อมูลทะเบียนสมาชิก (`MemberRoster`) 
3. **ระบบตอบคำถาม (Knowledge Base):** ตอบคำถามอัตราดอกเบี้ย/สวัสดิการจากตาราง `KnowledgeEntry` (Fallback ไปที่ `lib/knowledge.ts`) 
4. **ความปลอดภัย:** กรองลิงก์ที่ไม่ใช่โดเมนสหกรณ์ (`nktscoop.com`) ออกจากคำตอบทั้งหมดผ่าน `stripDisallowedLinks`
5. **ค้นหาเลขสมาชิก:** เมื่อสมาชิกถามหาเลขสมาชิกของตัวเอง บอทต้องขอยืนยันตัวตนก่อนเสมอ 3 อย่าง (ชื่อ-นามสกุล, เลขประจำตัวประชาชน 13 หลัก, เบอร์โทรที่ลงทะเบียนไว้) แล้วตรวจกับ `MemberRoster` แบบ exact match (`lib/memberLookup.ts` — เทียบเลขบัตร/เบอร์โทรแบบตัดขีด/เว้นวรรคออกก่อน, เทียบชื่อแบบเดียวกับ `nameMatch.ts`) ไม่ตรงครบทั้ง 3 อย่างจะไม่บอกเลขสมาชิกเด็ดขาด และไม่บอกว่าช่องไหนผิดเพื่อกันการเดา — ถ้าสมาชิกโทรมาแจ้งเปลี่ยนเบอร์โทรหรือแจ้งว่าเลขบัตรที่มีอยู่ผิด เจ้าหน้าที่แก้ไขได้จากหน้า dashboard โดยตรง (แท็บ "สมาชิก LINE" > "แก้ไขข้อมูลยืนยันตัวตนสมาชิก", `components/MemberContactPanel.tsx` / `app/api/member-roster/`) — จงใจไม่เปิดให้สมาชิกแก้เองผ่านบอท เพราะ 2 ช่องนี้คือช่องที่ใช้ยืนยันตัวตนก่อนบอกเลขสมาชิก ถ้าให้แก้เองได้จะเท่ากับให้ใครก็ตาม "รีเซ็ต" ข้อมูลยืนยันตัวตนเป็นค่าที่ตัวเองคุมได้ ตัวตรวจรูปแบบ (13 หลัก / เบอร์ 9-10 หลัก) ใช้ร่วมกับสคริปต์ `import-national-id-phone.ts` ผ่าน `lib/identityFormat.ts`

### 🎯 การจับคู่ผู้รับผิดชอบเรื่องกู้เงิน (Loan Routing)
คำขอที่แผนก "สินเชื่อ" หาผู้รับตามลำดับความสำคัญ (`resolveForwardTargets` ใน `lib/agent/forwarding.ts`, precedence อยู่ใน `lib/loanRouting.ts`):
1. **รหัสผู้รับผิดชอบรายบุคคล** (`MemberRoster.responsibleCode` จับคู่ `ResponsibleContact.code`) — วิธีหลัก แม่นยำกว่าเพราะเป็นรหัสสั้นๆ ไม่ใช่ข้อความอิสระ จัดการได้จากหน้า dashboard ("ผู้รับผิดชอบเรื่องกู้เงิน") โดยตรง หรือนำเข้าเป็นชุดด้วย `import-responsible-contacts.ts`
2. **ชื่อหน่วยงาน** (`MemberRoster.unitName` จับคู่ `LoanDistrictContact.unitName` แบบตรงตัว) — fallback สำรอง นำเข้าด้วย `import-loan-contacts.ts`
3. **`LINE_FORWARD_LOAN_ID`** — fallback สุดท้ายเมื่อไม่มีทั้งสองข้อบน

### 🏢 การส่งต่อคำขอตามแผนก (Department Routing)
แผนกอื่นที่ไม่ใช่สินเชื่อ (เงินฝาก, สารสนเทศ, สวัสดิการ, นิติการ, บัญชี, ฌาปนกิจ, บริหารสำนักงาน/ธุรการ, อื่นๆ — รายชื่ออยู่ใน `lib/departments.ts`) ส่งต่อผ่านตาราง `DepartmentContact` (precedence อยู่ใน `lib/departmentRouting.ts`):
1. **ส่งหาเจ้าหน้าที่ทุกคนที่ตั้งไว้ในแผนกนั้นพร้อมกัน** (broadcast) — กันคำขอตกหล่นเพราะไม่ต้องรู้ว่าใครอยู่เวร จัดการเจ้าหน้าที่ต่อแผนกได้จากหน้า dashboard โดยตรง ไม่ต้อง deploy ใหม่
2. **`LINE_FORWARD_GENERAL_ID`** — fallback เมื่อแผนกนั้นยังไม่มีเจ้าหน้าที่ตั้งไว้เลย

ถ้าส่งสำเร็จอย่างน้อย 1 คน ระบบจะถือว่าคำขอ "ส่งต่อสำเร็จ" (แจ้งสมาชิกตามนั้น) แต่จะบันทึกใน `ServiceRequestLog.forwardedTo` ว่าใครล้มเหลวด้วย เผื่อเจ้าหน้าที่บางคน LINE ID ใช้ไม่ได้แล้วให้เจ้าของระบบสังเกตเห็นและแก้ไข

### 💰 การแจ้งเตือนธุรกรรม (Transaction Forwarding)
ทุกครั้งที่บันทึกธุรกรรมสำเร็จ (`finalizeTransaction`) ระบบจะแจ้งเจ้าหน้าที่แผนกที่รับผิดชอบพร้อมสลิปแนบ (best-effort — ไม่บล็อกการบันทึกธุรกรรมแม้ส่งไม่สำเร็จ) โดยจับคู่หมวดหมู่ธุรกรรม ↔ แผนกไว้ที่ `lib/categoryDepartments.ts`:

| หมวดหมู่ | แผนก |
| --- | --- |
| ฝากเงิน | เงินฝาก |
| ซื้อหุ้น, ชำระหนี้ | สารสนเทศ |
| ชำระเก็บไม่ได้รายเดือน | สินเชื่อ (ใช้ per-member routing เดียวกับคำขอสินเชื่อ) |
| ชำระประกัน, สสอค, สสชสอ, สสสก, สสสท | สวัสดิการ |
| ชำระฌาปนกิจ, สสค | ฌาปนกิจ |

ผลการส่ง (`forwardedTo`/`forwardStatus`) บันทึกไว้ที่แถว `Expense` เอง ไม่มีตาราง log แยก

### 💻 แดชบอร์ดเจ้าหน้าที่ (Dashboard)
เข้าถึงผ่านหน้าแรก (`/`) ป้องกันด้วย **Basic Auth** (`middleware.ts`) แบ่งเป็นแถบ (`components/Tabs.tsx`) เพื่อไม่ให้ต้องเลื่อนหน้ายาว — แต่ละแถบโหลดข้อมูลเฉพาะตอนเปิดดูครั้งแรก:
* **ธุรกรรม:** ยอดรวมประจำเดือน กราฟแยกหมวดหมู่ ตัวกรอง/คิวตรวจสอบ (`memberVerified = false` เพื่อผูก LINE ID เข้ากับทะเบียนสมาชิก) รายการธุรกรรม ส่งออก CSV (BOM Excel) และฟอร์มบันทึกโดยเจ้าหน้าที่
* **คำขอบริการ:** ทะเบียนคำขอส่งต่อเอกสารประกอบ พร้อมรูปถ่ายและเบอร์ติดต่อ พร้อมปุ่ม "เพิ่มเข้าทะเบียน" สำหรับสมาชิกที่ยังไม่ยืนยันตัวตน
* **ผู้รับผิดชอบ:** จัดการรหัสผู้รับผิดชอบฝ่ายสินเชื่อ ↔ LINE UserId (แทนการแก้ผ่านไฟล์ Excel) และเจ้าหน้าที่ผู้รับคำขอของแต่ละแผนกอื่น ได้เองจากหน้านี้
* **ฐานความรู้:** แก้ไขค่าบริการ/ดอกเบี้ย/สวัสดิการในตาราง `KnowledgeEntry` ได้โดยตรง (Cache 60 วินาที)
* **สมาชิก LINE:** ค้นหา/แบ่งหน้ารายชื่อสมาชิกที่เคยทักบอท พร้อมเครื่องมือ "ล้างข้อมูลทดสอบ" ลบข้อมูลทุกตารางของเลขสมาชิกทดสอบ (เช่น TEST9999) ในคลิกเดียว แสดงจำนวนแถวให้ตรวจสอบก่อนยืนยันลบ

## ⚙️ การตั้งค่าสภาพแวดล้อม (Environment Variables)
คัดลอก `.env.example` ไปเป็น `.env` และตั้งค่าตัวแปรหลัก:
* `DATABASE_URL`: Connection string ของ PostgreSQL
* `LINE_CHANNEL_SECRET` & `LINE_CHANNEL_ACCESS_TOKEN`: คีย์ของ LINE Messaging API
* `ANTHROPIC_API_KEY`: API Key ของ Claude (Anthropic)
* `BLOB_READ_WRITE_TOKEN`: (Optional) token ของ Vercel Blob Store สำหรับสำรองรูปสลิป/เอกสาร — สร้างแบบ **Private** (Vercel ไม่มี Public store ให้เลือกแล้ว) เก็บ/อ่านรูปผ่าน `/api/blob/[...path]` (proxy ในโค้ด ยกเว้นจาก Basic Auth เพราะ LINE ต้องดึงรูปได้โดยไม่มี credential) — ถ้าไม่ตั้งค่า ระบบจะยังทำงานได้ปกติ แค่ไม่มีรูปแนบไปกับข้อความที่ส่งต่อให้เจ้าหน้าที่
* `LINE_FORWARD_LOAN_ID` & `LINE_FORWARD_GENERAL_ID`: LINE ID เจ้าหน้าที่สำหรับส่งต่อคำขอ (ตั้ง `LOG_EVENT_SOURCES=1` ชั่วคราวเพื่อหา ID ใน Log ได้)

## 🛠️ คำสั่งการใช้งาน (Commands)

```bash
# ติดตั้ง Library
npm install

# การจัดการฐานข้อมูล (Prisma)
npx prisma migrate dev       # ตอนพัฒนาในเครื่อง
npx prisma migrate deploy    # ตอนขึ้น Production

# นำเข้าข้อมูลอ้างอิงจาก Excel (ไม่บังคับ)
npx tsx scripts/import-org-data.ts <path-to-file.xlsx>              # นำเข้าสมาชิก/หน่วยงาน (รวมรหัสผู้รับผิดชอบจากคอลัมน์ H, เลขบัตรประชาชนจากคอลัมน์ F, เบอร์โทรจากคอลัมน์ G)
npx tsx scripts/import-responsible-contacts.ts <path-to-file.xlsx>  # นำเข้ารหัส -> LINE UserId ผู้รับผิดชอบ (วิธีจับคู่หลัก)
npx tsx scripts/import-loan-contacts.ts <path-to-file.xlsx>         # นำเข้าผู้รับผิดชอบตามชื่อหน่วยงาน (fallback สำรอง)
npx tsx scripts/import-national-id-phone.ts <path-to-file.xlsx>     # เติมเลขบัตรประชาชน/เบอร์โทรให้สมาชิกที่มีอยู่แล้วเท่านั้น (คอลัมน์ F/G) — ใช้เมื่อไฟล์หลักยังไม่มี 2 คอลัมน์นี้ตอน import-org-data.ts ครั้งแรก ไม่แตะ field อื่นเลย
npx tsx scripts/inspect-responsible-codes.ts <path-to-file.xlsx>    # ตรวจโครงสร้างไฟล์ก่อน import จริง (read-only)

# รันระบบและตรวจสอบ
npm run dev                  # รัน Local server (ตั้ง Webhook ไปที่ /api/line/webhook)
npx tsc --noEmit             # ตรวจสอบ TypeScript
npm test                     # รัน Unit Tests (Vitest)
npm run build                # บิลด์ระบบสำหรับ Deploy
